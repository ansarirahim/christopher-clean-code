Regarding I2C addresses:



The DA7281 has 2 possible I2C addresses determined by a hardware pin (ADDR):
- 0x4A when ADDR pin is tied to GND
- 0x4B when ADDR pin is tied to VDD



So for 4 devices on the same I2C bus, you'll need to split them across 2 separate I2C buses (2 devices per bus). The nRF52833 has TWI0 and TWI1, so this works perfectly:
- TWI0: 2 devices (addresses 0x4A and 0x4B)
- TWI1: 2 devices (addresses 0x4A and 0x4B)



My HAL will support this - each device instance will specify which TWI bus and which address to use.



Regarding your question about my existing code:



The DA7281 HAL in my GitHub repo was based on the standard nRF5 SDK v17.1.0 with FreeRTOS. I used Nordic's nrf_drv_twi driver for I2C communication and standard FreeRTOS primitives (mutexes for thread safety, vTaskDelay for timing).



I'll adapt this to work with the Qorvo SDK v1.1.1. Since Qorvo's SDK is built on top of nRF5 SDK, the I2C driver interface should be similar. I'll verify this when I set up the SDK environment in the next day or two.

Below is a summary of the issues I have identified while debugging the HAL. All points are based directly on the DA7281 datasheet (v3.1) and the current implementation in the project.



1. Incorrect register for actuator type:
The code writes the motor type to TOP_CFG2.
The datasheet defines actuator type in TOP_CFG1 bit 5, not in TOP_CFG2.
All writes related to selecting LRA/ERM must be moved to TOP_CFG1 bit 5.



2. Incorrect register for setting operation mode: (Note I fixed this in recent commit 585b4ee)
The code writes the operating mode into TOP_CFG1[2:0].
OPERATION_MODE is actually in TOP_CTL1 (address 0x22), bits [2:0].
Writing mode to TOP_CFG1 has no effect. The driver must write to TOP_CTL1.



3. All actuator-parameter calculations (in da7281_configure_lra) use incorrect formulas:
- LRA_PER_H/L: should follow the datasheet equation using 1333.32e-9 scaling, not 1/f.
- V2I_FACTOR: uses a simplified impedance 1.5, but datasheet requires (Z (IMAX + 4)) / 1.6104.
- ACTUATOR_NOMMAX and ACTUATOR_ABSMAX: code uses 11.7 mV LSB, but datasheet defines 23.4 mV per LSB.
- ACTUATOR_IMAX: code uses 7.8125 mA per LSB, but datasheet defines (I_mA - 28.6) / 7.2.
All formulas need correction to match the datasheet values and scaling.



4. Incorrect registers used for override-mode drive:
The code writes override settings into GEN_CFG2 and OVERRIDE_AMP.
The datasheet defines no GEN_CFG2 register.
Override amplitude must be written to TOP_CTL2 (address 0x23) using field OVERRIDE_VAL.
Override mode must be entered by writing OPERATION_MODE = 1 (DRO) in TOP_CTL1 (address 0x22).
As implemented, the override path writes to non-existent registers and cannot work.



5. Incorrect or non-existent registers still present in the register header: (Note some of these fixed in recent PR)
Some register #defines correspond to names or addresses that do not exist in the datasheet,
including:
- DA7281_REG_ACTUATOR_TYPE (0x0B)
- Any "GEN_CFGx" registers (0x90‚Äì0x95 range in header does not exist in datasheet)
These defines must be removed or renamed to match Table 20 exactly.



6. Bit fields mislabeled as registers:
Some entries in the header define independent registers for what are actually bit fields inside
existing registers (for example entries at addresses already assigned to LRA_PER or ACTUATOR values).
These duplicates must be removed so that each hardware register address appears exactly once.



7. Missing required fault-handling steps in the HAL:
Before attempting to drive, the datasheet requires clearing fault bits by writing ones to
IRQ_EVENT1 (address 0x03). The HAL does not implement this.
After any failure, the HAL should read IRQ_STATUS1 to detect actuator/BEMF/frequency faults.
Without these fault clears, the device can remain in a non-operational state.



These items represent the changes required for correctness according to the DA7281 v3.1 datasheet that I have identified thus far.
